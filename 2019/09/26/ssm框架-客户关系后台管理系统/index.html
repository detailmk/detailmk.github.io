用户模块

登录功能

  • 流程:
    • 后台

  1. 输入用户名密码 发送请求

  2. 后台接收参数校验(非空校验和唯一性校验)

  3. 通过用户名查询用户,查看密码是否匹配

  4. 返回登录结果 前台接收信息

    • 前台

  5. 引入jq,js

  6. 前台获取表单参数

  7. 校验参数非空

  8. 发送ajax请求

登录后台userService代码优化

—->通过assertutil工具类抛异常简化业务层代码

controller层

用户登录状态

技术:采用cookie

  • 流程

用户登录成功后,把用户id进行加密处理存入cookie中

后台负责加密userId存到存储信息中

前台取到数据进行存储

修改密码功能

  • 流程
  1. 接收前台用户旧密码新密码
  2. 参数校验(非空)
  3. 校验旧密码是否正确
  4. 正确用新密码修改 修改成功返回登录页面

优化用户模块(全局异常处理,控制层重复代码处理)

  • 流程

处理异常,如果页面报错,统一返回报错页面

​ 如果数据报错,统一返回报错的JSON数据

  • 全局处理异常思路

通过方法上是否有@ResponseBody注解来区分

视图的话设置错误返回视图

json的话通过流返回json错误信息

控制层重复代码处理1. 返回视图 2. 对于设置返回json数据的处理

注册功能

  • 流程
  1. 接收前台用户注册信息
  2. 对前台接收参数进行校验(**用户名唯一性检验)
  3. 返回校验结果,成功执行用户添加操作并跳转到登录页面,失败返回校验失败的原因

非法登录拦截

  • 流程

拦截器获取cookie中的userid

如果userid存在,则则放行

如果不存在跳转登录页面

  • 出现bug

样式丢失–没配置放行文件

类型转换异常–若未登录返回

error视图–没有uri请求地址

防止错误页面嵌套–window.parent.location.href

营销模块

营销机会模块

描述:对市场营销机会进行编辑操作

分页查询

  • 流程

后台:mapper->service->controller

添加

正常添加操作,对必要参数进行校验

主要处理指派人 通过三表查询显示指派人下拉框

前台表单获取指派人姓名交由后台做添加或者更新处理

更新

更新和添加使用同一个弹出框里的表单

在业务层添加区分更新还是添加的逻辑(是否存在id)

  • 共用一个窗口出现问题

执行完更新操作,点击添加,添加框出现数据回显bug

–>解决方式:对话框执行关闭时就执行清空表单操作

删除

  • 流程

后台base假删除通过修改有效值属性

前台:判断是否选中单个或多个记录

​ 通过发送ajax请求进行单条或多条记录删除

优化增删改查

由于增删改前台代码大多重复,进行封装js代码

营销计划模块

描述:对营销机会里中的客户进行营销操作

显示两块内容

第一块显示需要采取营销计划的客户(通过作用域存储到后台查询用户信息),

第二块显示具体营销计划内容(通过ajax请求对计划进行编辑)

在编辑营销计划时修改营销状态

出现bug:在一个从未开发的客户中执行开发计划,开发状态还是为未开发状态

解决:完善业务层判断,当执行一次添加计划操作,判断状态,如果为未开发状态则修改,如果不是开发状态,则不用修改

客户模块

流失客户模块

客户流失规则定义:
客户自创建超过六个月以来未与企业产生任何订单以及客户最后下单日期超过六个月的客户定位流失客户

查询流失客户信息

条件:客户创建超过六个月没有下过订单,

​ 距离最后下单超过六个月

流程

根据流失客户定义查询流失客户

查询的客户存在容器里批量添加到流失客户中

获取客户id,批量更新流失客户的流失状态

权限模块

权限管理(RBAC)-基于角色的权限管理系统(*****)
1.表结构
用户 角色 菜单(资源)
用户-角色:多对多
角色-菜单:多对多

t_user   用户表
t_user_role （id,user_role,role_id,is_valid,create_date,update_date）
t_role   角色表
t_permission(id,role_id,module_id,aclValue,is_valid,create_date,update_date)
t_module  资源表

三种对应关系(关系库)
    一对一   一对多  多对多

2.功能实现
用户管理模块
查询、用户添加、更新、删除用户、
关联角色(添加用户关联角色、更新用户关联角色，删除用户级联删除用户角色记录):
后台: 添加用户|更新用户:删除原始的用户角色记录(如果存在时)，加入新的角色记录(t_user_role)
前台:使用下拉框(支持多选)
角色管理模块
查询、添加、更新、删除(级联权限记录)
授权:给角色配置操作菜单的权限(t_permission)
前端:
菜单资源展示(zTree插件 展示父子关联表数据)
获取选中的菜单记录(定义zTree callback 事件+方法)
原始权限的回显(zTree 数据属性-checked=true)

后端:删除原始的权限记录(如果存在时)，加入新的权限记录(roleId,mid[])

认证:对角色操作的菜单做校验
    前端菜单显示的控制:
            用户登录成功 根据用户分配角色 查询角色的权限(List<String>) 放入session
            视图中判断菜单权限码是否被session  中存放的权限列表包含  <#if permissions?seq_contains("10")></#if>

                 后端方法控制:AOP+自定义注解
                      自定义注解配置权限码
                      AOP 拦截指定注解 借助环绕通知(目标方法执行前判断拦截到的注解的权限码  session 中存在该权限码 方法放行 不存在 方法抛出异常)
     资源管理模块
          查询、添加、修改、删除
             二、三级菜单:  菜单的添加|修改(需要关注上级菜单)  
             资源名  权限码  唯一校验
             删除资源 级联权限(t_permission)
